## Setup

| Parameter | Value |
|-----------|-------|
| Volume | recovar assets/vol0000.mrc, 128^3 |
| Particles | 5000 |
| Pixel size | 4.25 A |
| Noise level | 0.1 (white Gaussian) |
| Contrast/noise scale std | 0 (uniform) |
| Rotation grid | HEALPix level 3 (36,864 orientations) |
| Translation grid | +/-3 px, step 1 px (29 translations) |
| Max iterations | 10 (recovar), auto-refine (RELION) |
| Initial volume | Low-pass filtered GT (r<=5 shells) |
| GPU | NVIDIA A100-SXM4-80GB |
| Slurm job (RELION) | 6352420 (NUFFT data), 6352949 (RELION data) |

## Data Generation

Each code was tested on images generated by a compatible forward model:

- **recovar**: Images generated by the NUFFT simulator (`disc_type="nufft"`), which
  computes exact Fourier slices via non-uniform FFT. Reconstruction uses
  `linear_interp` (trilinear interpolation in Fourier space), matching RELION's
  kernel.

- **RELION**: Images generated by `relion_project --ctf --pad 2`, which uses RELION's
  own trilinear interpolation forward model. Additive white Gaussian noise was added
  post-projection (noise std = 0.1 * image std).

Both datasets share the same ground truth volume, CTF parameters, and Euler angles
(verified round-trip `R_to_relion` / `R_from_relion` to machine precision).

## Convention Verification

The Euler angle convention was verified at two levels:

1. **Round-trip test**: `R_from_relion(R_to_relion(R))` recovers R to 1e-15 for 100
   random rotations.

2. **Projection test**: An asymmetric 3-blob volume was projected at (rot=0, tilt=0,
   psi=0) using both RELION (`relion_project`) and numpy sum-along-Z. CC = 0.9999,
   confirming the volume and angle conventions are consistent.

3. **Cross-code images are NOT compatible**: Despite matching conventions, images
   produced by NUFFT and by `relion_project` have CC ~ 0 for the same angle and
   volume. This is because NUFFT operates in continuous Fourier space while RELION
   uses discrete trilinear interpolation -- the two produce fundamentally different
   pixel values. Each code's refinement pipeline is self-consistent with its own
   forward model but cannot process images from the other's forward model.

## Analysis

### Why RELION Failed to Converge

RELION's auto-refine showed angular changes of 70-76 degrees even in the final
iterations, indicating it never found consistent orientations. The self-reported
resolution oscillated between 14-108 A. Two factors contributed:

1. **Low SNR at the working resolution**: With only 5000 particles at 128^3 and
   noise_level=0.1, the per-image SNR in the resolution range where orientation
   search happens is marginal. RELION's significance pruning may be discarding too
   many poses.

2. **Convergence instability**: RELION's resolution jumped from 13.95 A back to
   108.8 A at iteration 7, then partially recovered. This is a known failure mode
   of RELION's auto-refine when the signal is weak.

### Why recovar Succeeded

recovar's dense E-step evaluates ALL rotation/translation combinations (36,864 x 29
= 1,069,056 poses per image per half-set) without pruning. This brute-force approach
is more robust for ab-initio-like scenarios. The Fourier windowing (current_size
control) provides resolution-dependent focusing that prevents overfitting to noise at
high frequencies.

## Caveats

1. This comparison is on simulated data with known poses for initialization. A fairer
   comparison would use ab-initio reconstruction (e.g., `relion_refine --inimodel`
   without pose priors).

2. RELION was not tuned for this specific dataset (e.g., particle diameter, initial
   model resolution, regularization). Better results might be achievable with
   different parameters.

3. The datasets are NOT cross-compatible: NUFFT-generated images cannot be
   reconstructed by RELION and vice versa, because the discrete forward models are
   sufficiently different.
